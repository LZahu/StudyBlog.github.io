java-类加载器与双亲委派模型

发表于 2021-07-25 更新于 2021-07-29 分类于 java
介绍类加载器的双亲委派模型，并实际操作下如何打破双亲委派模型。

1. 类加载器

一般而言，类文件的生命过程可以简单分为：加载、连接、初始化、使用和卸载。对于第一个步骤：类加载而言，它要求是通过一个类的全限定类名（包名+类名）来获取该类的二进制字节流。在jvm中，这种获取（或者说加载）的方法并没有被定死了，而是允许应用程序自己决定如何去获取所需的类，那么，这样一种可自定义的方法（代码）被称为“类加载器”。

对于任意一个类，都必须由加载它的类加载器和这个类本身一起共同确立其在Java虚拟机中的唯一性，每一个类加载器，都拥有一个独立的类名称空间。

也就是说，比较两个类是否“相等”，只有在这两个类是由同一个类加载器加载的前提下才有意义，否则，即使这两个类来源于同一个Class文件，被同一个Java虚拟机加载，只要加载它们的类加载器不同，那这两个类就必定不相等。

通俗地说，就是“拿来吧你”！

2. 双亲委派模型

2.1. 类加载器分类

  1. 启动类加载器（Bootstrap Class Loader）
    • 用C语言实现，嵌套在jvm内部，不继承于 java.lang.ClasssLoader ，没有父加载器
    • 加载java的核心库（JAVA_HOME/jre/lib/rt.jar、resources.jar 或者 sun.boot.class.path路径下的内容），提供jvm自身需要的类
    • 出于安全考虑，Bootstrap只加载包名为java、javax、sun等开头的类
  2. 扩展类加载器（Extension Class Loader）：
    • java语言编写，派生于 java.lang.ClasssLoader
    • 它负责加载 <JAVA_HOME>\lib\ext 目录中，或者被 java.ext.dirs 系统变量所指定的路径中所有的类库
  3. 应用程序类加载器（Application Class Loader）：
    • java语言编写，派生于 java.lang.ClasssLoader
    • 它负责加载 用户类路径（ClassPath） 上所有的类库
    • 如果应用程序中没有自定义过自己的类加载器，一般情况下这个就是程序中默认的类加载器。

2.3. 双亲委派模型介绍

正如第一张图片中显示的，比较有意思的是，在实际加载某一类的时候，是一种先向上溯源，再向下推进的过程！

具体地说，如果一个类加载器收到了类加载的请求，它首先不会自己去尝试加载这个类，而是把这个请求委派给上层加载器去完成，每一个层次的类加载器都是如此，因此所有的加载请求最终都应该传送到最顶层的启动类加载器中，只有当上层加载器反馈自己无法完成这个加载请求（它的搜索范围中没有找到所需的类）时，下层加载器才会尝试自己去完成加载。

我以为，“双亲委派模型”这个名字不太合适，哪里有双亲了？倒不如叫“递推模型”。。。

2.4. 优势

  1. 避免类的重复加载
  2. 保护程序的安全，防止核心API被恶意篡改（沙箱安全机制）

2.5. loadClass( )方法

实现双亲委派的代码都集中在java.lang.ClassLoader的loadClass()方法之中！

总的来说，分为4步：

  1. 先检查该类有没有被加载过
  2. 若没有加载则调用父加载器的 loadClass() 方法进行加载
  3. 若父加载器为空则默认使用启动类加载器作为父加载器，并调用它的 loadClass() 方法
  4. 如果父类加载失败，抛出 ClassNotFoundException 异常后，再调用自己的 findClass() 方法进行加载
