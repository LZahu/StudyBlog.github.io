1.2. Http Code 状态码

分为5种不同情况：

  1. 1xx状态码对应该请求已被接收，需要继续处理。HTTP/1.0协议中没有定义任何1xx状态码，所以除非在某些试验条件下，服务器禁止向此类客户端发送1xx响应
  2. 2xx状态码对应的该请求已成功被服务器接收
  3. 3XX状态码对应该请求需要重定向。需要客户端采取进一步的操作才能完成请求
  4. 4xx状态码对应客户端请求出错
  5. 5xx状态码对应服务端出错

常见状态码：

状态码 描述
202 请求成功
301 资源（货网页等）被永久转移到其他URL
404 请求的资源（或网页等）不存在
500 内部服务器错误

1.3. http简介

Http全称是超文本传输协议，是因特网上应用最为广泛的一种网络传输协议。是一个基于 TCP/IP 通信协议来传递数据（HTML文件，图片文件，查询结果等）。从层次的角度看，HTTP是面向事务的应用层协议。

HTTP协议工作在客户端-服务端架构上。浏览器作为HTTP客户端通过URL向HTTP服务端（即WEB服务器）发送所有请求。WEB服务器根据接收到的请求后，向客户端发送响应信息。也就是说：肯定是先从客户端开始建立通信，服务器端在没有接收到请求之前不会发送响应。

1.3.1. 关于http的三点注意事项

  1. HTTP是无连接：无连接的含义是限制每次连接只处理一个请求。服务器处理完客户的请求，并收到客户的应答后，即断开连接。采用这种方式可以节省传输时间。
  2. HTTP是媒体独立的：这意味着，只要客户端和服务器知道如何处理的数据内容，任何类型的数据都可以通过HTTP发送。客户端以及服务器指定使用适合的MIME-type内容类型。
  3. HTTP是无状态：HTTP协议是无状态协议。无状态是指协议对于事务处理没有记忆能力。缺少状态意味着如果后续处理需要前面的信息，则它必须重传，这样可能导致每次连接传送的数据量增大。另一方面，在服务器不需要先前信息时它的应答就较快。

1.4. http报文结构

http报文分为请求报文和响应报文

这两种报文都是有三个部分组成：开始行、首部行、实体主体

  • 开始行：用于区分请求报文还是响应报文。在请求报文中的开始行叫做**请求行(Request-Line)，而在响应报文中的开始行叫做状态行(Status-Line)**。开始行的三个字段之间都以空格分隔开，最后的“CR”和“LF”分别代表回车和换行
  • 首部行：用来说明浏览器、服务器或报文主体的一些信息。首部可以有好几行，但也可以不使用。在每一个首部行中都有首部字段名和它的值，每一行在结束的地方都要有”回车“和”换行“。整个首部行结束时，还有一空行将首部行和后面的实体主体分开。
  • 实体主体：在请求报文中一般都不用这个字段，而在响应报文中也可能没有这个字段。

1.4.1. 请求报文

请求报文中的请求行：

请求行只有三个内容，即方法，请求资源的URL，以及HTTP的版本。“方法”是面向对象技术中使用的专门名词。所谓“方法”就是对所请求的对象进行的操作，因此这些方法实际上也就是一些命令。因此，请求报文的类型是由它所采用的方法决定的。

1.7. https简介

HTTPS （全称：Hyper Text Transfer Protocol over SecureSocket Layer），是以安全为目标的 HTTP 通道，在HTTP的基础上通过传输加密和身份认证保证了传输过程的安全性。

HTTPS 在HTTP 的基础下加入 SSL 层，HTTPS 的安全基础是 SSL，因此加密的详细内容就需要 SSL。 HTTPS 存在不同于 HTTP 的默认端口及一个加密/身份验证层（在 HTTP与 TCP 之间）。这个系统提供了身份验证与加密通讯方法。它被广泛用于万维网上安全敏感的通讯，例如交易支付等方面.

1.8. https加密过程

HTTPS 在内容传输的加密上使用的是对称加密（共享密钥），非对称加密（公开密钥）只作用在证书验证阶段。

证书验证阶段

  1. 浏览器发起https请求

  2. 服务端返回https证书（包含了公钥）

  3. 客户端验证证书是否合法，如果不合法则提示警告

数据传输阶段

  1. 验证证书合法性（这一部分由SSL/TLS完成）。当证书验证合法后，在本地生成随机数

  2. 通过公钥加密随机数，并把加密后的随机数传输到服务端

  3. 服务端通过私钥对随机数进行解密

  4. 服务端通过客户端传入的随机数构造对称加密算法，对返回结果内容进行加密后传输

  5. 客户端根据本地存储的随机数进行解密

1. 初级优化分析

红色的代表可能会发生高并发的操作。

2. 详情页缓存

通过CDN缓存静态资源，避免多次刷新导致崩溃。

  1. 动静态数据分离：详情页静态资源是部署在CDN节点中，也就是说访问静态资源或者详情页是不用访问系统的。
  2. 用户点击秒杀操作的按钮之后，这个按钮会变灰，禁止重复提交，这样的话，也可以降低系统的一些压力。

3. 秒杀接口地址缓存优化

和详情页不同，此时无法使用CDN，因为秒杀接口是动态变化的，而CDN适合的请求资源是不易变化的。

此时，可以使用redis缓存存储热门的商品信息，这样的话，可以避免直接去MySQL数据库中查找数据，在高并发的情况下，redis的qbs可以达到10w，因此，使用redis缓存可以扛住很大的并发量。当然，还可以考虑redis集群，这样的话，并发量还可以再进一步提升。

具体的操作流程，可以对应到上一篇博客中的 exportSeckillUrl 函数，在具体使用redis缓存的过程中，首先是先在redis中查找商品，如果没找到，再去MySQL数据库中查找，如果查找到了，一边返回商品信息，一边将该商品信息放入redis中，以便下一次的快速查询。

4. 秒杀操作优化

在上一篇博客的 executeSeckill 函数中，执行秒杀操作的数据库更新顺序是：插入购买明细 -> 减库存，但是，实际上，我们首先想到的应该是反过来的。

这样的方式，可能存在的问题有三个：

  1. 在MySQL数据库中，默认是行级锁，也就是说在执行第一个update的时候就会持有行级锁，并且一直到commit才会释放掉，在这个过程中，所有对这个商品操作的事务都会被阻塞。
  2. 存在网络延迟，这是无法避免的，而且这种网络延迟是双向的，即发送请求会产生延迟，返回结果也会产生延迟。
  3. 可能会发生GC，那么可能就会发生stop the world，意味着所有操作全部暂停。

因此，按照这个顺序来执行的一段时间就可以分为三部分：MySQL执行时间 + 网络延迟 + GC时间。

简单点，单从MySQL的执行顺序上考虑，把update和insert调换一下顺序：

这样做可以减少时间的原因是：插入是可以并行的，而更新由于会加行级锁是串行的。那么，这样就可以减少锁的持有时间，一定程度上提高并发度。

4.2. 进一步优化

1. 业务分析

业务的重点就是对库存的操作，而高并发的操作主要是集中于用户端，所做的操作包括两部分：

  1. 减库存
  2. 记录购买明细

1.1. 用户对库存的具体业务

  1. 减库存

    在商品信息表中，将对应商品的库存量减1

  2. 记录购买明细

    在本项目中，购买明细指的是：商品id、用户id、购买状态、购买时间。

在完整的秒杀操作流程中，这两步组成了用户的购买行为，缺一不可。

1.2. 事务

正是因为上述所说的，这些操作缺一不可，因此，需要用数据库的事务来完成这些操作！在MySQL数据库中，它内置的事务机制，可以准确的帮我们完成减库存和记录用户购买明细的过程。

如果没有事务支持，这些操作发生了错误，会导致什么后果？

  1. 若是用户成功秒杀商品我们记录了其购买明细却没有减库存。导致商品的超卖
  2. 减了库存却没有记录用户的购买明细。导致商品的少卖

2. 难点分析

如何高效地处理竞争？

在MySQL数据库中，会默认开启行级锁，比如此时正有一个事务正在对id为1的商品做操作：

  1. 开启事务
  2. 对商品的库存量减1
  3. 记录商品信息和用户购买信息
  4. 提交事务

只有等到这4步操作全部完成，行级锁才会释放。而此时其他所有针对id为1的商品的事务都会被阻塞，导致系统效率低下。

因此，项目的难点就在于如何高效地处理事务？如何高效地处理竞争？

7.2. 暴露秒杀接口

首先，exportSeckillUrl 函数做了一些优化的操作（在后面会具体讲到）。如果想拿到某个商品的秒杀地址，那么首先要进行如下步骤：

  1. 在redis缓存中查询是否有该商品？如果有，则可以直接得到该商品信息，如果redis缓存中没有，则在数据库中查找。
  2. 继上一步，再去数据库的商品信息表中查找，如果查找到该商品，则同样可以获得该商品的信息，并且，将该商品信息放入redis缓存，以便下次再缓存中直接查询。如果此时还是查不到，则说明不存在该商品，此时应该返回一个错误状态的秒杀接口类，说明该商品是不存在的。

那么，经过以上两步，如果拿到了该商品的详细信息，可以继续进行以下操作：

  1. 获得当前时间，目的是为了判断当前时间是否在秒杀时间段内。如果不在，比如还未开始秒杀，或者是秒杀已经结束了，同样返回一个错误状态的秒杀接口类，说明时间不对。
  2. 如果处于秒杀时间段内，那么需要获得随机的md5字段，作为盐值
  3. 根据md5，返回正确的秒杀接口类。

为什么要进行md5加密？

可以将md5理解为秒杀的凭证，也就是说，如果想要秒杀一个商品，仅仅知道该商品的信息还是不够的，还需要知道由这个商品信息而生成的md5，只有两者都符合，才能正确地执行秒杀。

从上述函数的过程发现，只有在获取到对应的商品信息，并处于秒杀时间段内，才可以暴露正确地秒杀接口！那么，此时可能会有三种情况：

  1. 如果有一些黄牛用户而已恶意猜测某些商品信息，比如随意输入某个商品id，但是，如果id错误，就会返回错误的秒杀接口，相当于第一道拦截；
  2. 如果黄牛用户猜对了id，但是呢，不处在秒杀时间段内，那么得到的还只是一个错误的秒杀接口，只是返回时间而已，没啥用，相当于第二道拦截；
  3. 就算黄牛用户猜对了id，猜对了时间，但是他没有匹配的md5，因为这个md5是根据商品信息经过特定方法生成的，所以很难匹配正确，那么这样的话，还是无法执行秒杀操作，避免了这些恶意操作的危害，相当于第三道拦截。

因此，md5就相当于校验值，起到了把关的作用！

java-G1垃圾收集器

发表于 2021-07-25 更新于 2021-07-29 分类于 java

1. 特点

G1 是一款主要面向服务端应用的垃圾收集器，开创了收集器面向局部收集的设计思路和基于 Region 的内存布局形式。

G1 收集器可以面向堆内存任何部分来组成回收集（Collection Set，一般简称 CSet）进行回收，衡量标准不再是它属于哪个分代，而是哪块内存中存放的垃圾数量最多，回收收益最大，这就是 G1 收集器的 Mixed GC 模式。

1.1. Region

G1 开创的基于 Region 的堆内存布局是它能够实现这个目标的关键。

虽然 G1 也仍是遵循分代收集理论设计的，但其堆内存的布局与其他收集器有非常明显的差异：G1 不再坚持固定大小以及固定数量的分代区域划分，而是把连续的Java堆划分为多个大小相等的独立区域（Region），每一个 Region 都可以根据需要，扮演新生代的Eden 空间、Survivor 空间，或者老年代空间。

收集器能够对扮演不同角色的 Region 采用不同的策略去处理，这样无论是新创建的对象还是已经存活了一段时间、熬过多次收集的旧对象都能获取很好的收集效果。

1.2. Humongous

Region 中还有一类特殊的 Humongous 区域，专门用来存储大对象。G1 认为只要大小超过了一个 Region 容量一半的对象即可判定为大对象。每个 Region 的大小可以通过参数-XX：G1HeapRegionSize 设定，取值范围为1MB～32MB，且应为2的 N 次幂。G1的大多数行为都把Humongous Region作为老年代的一部分来进行看待。

1.3. 可预测的停顿时间模型

G1 收集器之所以能建立可预测的停顿时间模型，是因为它将 Region 作为单次回收的最小单元，即每次收集到的内存空间都是Region大小的整数倍，这样可以有计划地避免在整个 Java 堆中进行全区域的垃圾收集。

更具体的处理思路是让 G1 收集器去跟踪各个 Region 里面的垃圾堆积的“价值”大小，价值即回收所获得的空间大小以及回收所需时间的经验值，然后在后台维护一个 优先级列表，每次根据用户设定允许的收集停顿时间（使用参数-XX：MaxGCPauseMillis 指定，默认值是200毫秒），优先处理回收价值收益最大的那些 Region。

2. 收集过程

  1. 初始标记 Stop The World

    仅仅只是标记一下 GC Roots 能直接关联到的对象，并且修改 TAMS 指针的值，让下一阶段用户线程并发运行时，能正确地在可用的 Region 中分配新对象。这个阶段需要停顿线程，但耗时很短，而且是借用进行 Minor GC 的时候同步完成的，所以 G1 收集器在这个阶段实际并没有额外的停顿。

  2. 并发标记

    从 GC Roots 开始对堆中对象进行可达性分析，递归扫描整个堆里的对象图，找出要回收的对象，这阶段耗时较长，但可与用户程序并发执行。

  3. 最终标记 Stop The World

    对用户线程做另一个短暂的暂停，用于处理并发阶段结束后仍遗留下来的最后那少量的 SATB 记录。

  4. 筛选回收

    根据各个 Region 的回收价值和成本进行排序，根据用户所期望的停顿时间（使用参数-XX：MaxGCPauseMillis 指定）来制定计划，把决定回收的那一部分Region存活的对象复制到空的 Region 中，再清理掉整个旧 Region 的全部空间。

3. 细节问题

3.1. 跨 Region 引用

使用记忆集避免全堆作为 GC Roots 扫描，但在 G1 收集器上记忆集的应用其实要复杂很多，它的每个 Region 都维护有自己的记忆集，这些记忆集会记录下别的 Region 指向自己的指针，并标记这些指针分别在哪些卡页的范围之内。

G1 的记忆集在存储结构的本质上是一种哈希表，Key 是别的 Region 的起始地址，Value 是一个集合，里面存储的元素是卡表的索引号。这种“双向”的卡表结构（卡表是“我指向谁”，这种结构还记录了“谁指向我”）比原来的卡表实现起来更复杂，同时由于 Region 数量比传统收集器的分代数量明显要多得多，因此 G1 收集器要比其他的传统垃圾收集器有着更高的内存占用负担。

3.2. 并发标记阶段与用户线程互不干扰

  1. 原始快照
  2. G1 为每一个 Region 设计了两个名为 TAMS（Top at Mark Start） 的指针，把 Region 中的一部分空间划分出来用于并发回收过程中的新对象分配，并发回收时新分配的对象地址都必须要在这两个指针位置以上。G1 收集器默认在这个地址以上的对象是被隐式标记过的，即默认它们是存活的，不纳入回收范围。

3.3. 建立可靠的停顿预测模型

G1 收集器的停顿预测模型是以衰减均值（Decaying Average）为理论基础来实现的，在垃圾收集过程中，G1收集器会记录每个 Region 的回收耗时、每个 Region 记忆集里的脏卡数量等各个可测量的步骤花费的成本，并分析得出平均值、标准偏差、置信度等统计信息。

4. G1和CMS的对比

  1. G1 可以指定最大停顿时间、分Region的内存布局、按收益动态确定回收集，但是，CMS仅仅是想获得最短的停顿时间
  2. G1 运作期间不会产生内存空间碎片，垃圾收集完成之后能提供规整的可用内存，但是，CMS会产生空间碎片，内存并不规整
  3. G1 和 CMS 都使用卡表来处理跨代指针，但 G1 的卡表实现更为复杂，而且堆中每个 Region，无论扮演的是新生代还是老年代角色，都必须有一份卡表，这导致 G1 的记忆集（和其他内存消耗）可能会占整个堆容量的20%乃至更多的内存空间；相比起来 CMS 的卡表就相当简单，只有唯一一份，而且只需要处理老年代到新生代的引用，反过来则不需要。由于新生代的对象具有朝生夕灭的不稳定性，引用变化频繁，能省下这个区域的维护开销是很划算的。
  4. G1 使用原始快照，相比增量更新能够减少并发标记和重新标记阶段的消耗，避免CMS那样在最终标记阶段停顿时间过长的缺点，但是在用户程序运行过程中确实会产生有跟踪引用变化带来的额外负担。
